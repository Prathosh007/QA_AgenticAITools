# Repo Documentation — Docs-First Navigation & Build Verification

> Shared across all platforms. Applied by all agents (Planner, Architect, Developer, Reviewer).

---

## Documentation Model

Every repository has structured documentation for AI agents:

```
<repo_root>/
├── .github/
│   └── copilot-instructions.md    ← Entry point (always read first)
└── docs/
    └── ai-agents/
        ├── README.md              ← Navigation hub, reading order
        ├── CODEBASE_MAP.md        ← File locations (task → file lookup)
        ├── BUILD_GUIDE.md         ← Build, test, and dependency commands
        ├── AGENT_GUIDE.md         ← Step-by-step task walkthroughs
        ├── GLOSSARY.md            ← Domain and platform term definitions
        ├── DOC_TEMPLATE.md        ← Documentation format standard
        └── DOC_UPDATE_PROTOCOL.md ← Which docs to update after code changes
    └── architecture/
        ├── ARCHITECTURE.md        ← Repo-wide architecture
        └── <project>/             ← Per-project architecture docs
```

Not every repo will have all files. Read what exists; skip what doesn't.

## Docs-First Rule (Mandatory — All Agents)

**Always navigate via documentation before searching code directly.**

### Reading Order

When working on any repo, follow this sequence:

1. **Read `.github/copilot-instructions.md`** — the entry point. It provides repo overview and points to relevant docs.
2. **Read `docs/ai-agents/README.md`** — navigation hub with reading order and project routing.
3. **Read the relevant docs based on your task:**

   | If you need to... | Read... |
   |---|---|
   | Find which files to look at | `docs/ai-agents/CODEBASE_MAP.md` |
   | Understand a domain term | `docs/ai-agents/GLOSSARY.md` |
   | Build, test, or verify | `docs/ai-agents/BUILD_GUIDE.md` |
   | Follow a task walkthrough | `docs/ai-agents/AGENT_GUIDE.md` |
   | Understand component architecture | `docs/architecture/ARCHITECTURE.md` or `docs/architecture/<project>/` |
   | Know what docs to update after changes | `docs/ai-agents/DOC_UPDATE_PROTOCOL.md` |

4. **Only then** read the specific source files the docs pointed you to.

### Search as Fallback Only

- **Do NOT** start with `grep_search`, `semantic_search`, or `file_search` to explore a repo
- **DO** use documentation to identify files first, then read those files directly
- **Fallback**: If docs don't cover what you need (gap or missing doc), then use search — and note the gap so it can be documented later

## Build Verification (Developer + Reviewer)

### Developer — Mandatory Build Execution

After implementing code changes, the Developer **must**:

1. Read `docs/ai-agents/BUILD_GUIDE.md` from the affected repo
2. Execute the appropriate build command (single-target or full build depending on scope)
3. Run linting if the repo's BUILD_GUIDE specifies lint commands
4. Run tests if the repo's BUILD_GUIDE specifies test commands
5. Capture and present the output (pass/fail, warnings, errors)

**Build failure blocks delivery.** Fix all build errors before presenting code at the checkpoint. Warnings should be addressed unless justified.

### Reviewer — Verification Commands Allowed

The Reviewer is **verification-only** (never modifies code or repository state) but **may run**:
- Build commands from BUILD_GUIDE.md to independently verify compilation
- Lint commands to verify style compliance
- Test commands to verify test results
- Verification commands: `codesign -dv`, `lipo -info`, `git status`, `git diff`

If the Developer's build output is clear and complete, the Reviewer does not need to re-run builds. Re-run only when output is ambiguous or incomplete.

## Documentation Updates — Per-Tag Strategy

**Do NOT update shared repo docs (CODEBASE_MAP, GLOSSARY, etc.) in feature branches.** This causes merge conflicts when multiple developers edit the same doc files in parallel.

### During Development (Feature Branch)

The Orchestrator **notes** what doc updates are needed in `.ai-docs/pending-doc-updates.md` — but does not touch shared docs. New feature-specific docs (new files under `docs/architecture/<feature>/`) can be created directly since they don't conflict.

```markdown
# Pending Documentation Updates
<!-- Generated by Orchestrator — processed by Doc Generator at tag time -->

**Branch:** feature/remote-shell-support
**Date:** 2026-03-13
**Task:** Add remote shell support to dcagentservice

## Updates Needed

### CODEBASE_MAP
- Add: `dcconfig/src/RemoteShellConfig.swift` — Remote shell configuration model

### GLOSSARY
- Add: "Remote Shell" — Server-initiated shell session via XPC

### AGENT_GUIDE
- Add: Walkthrough for adding a new remote command type

### BUILD_GUIDE
- No changes
```

This takes seconds and creates zero conflict risk.

### At Tag Time (Once Per Release Cycle)

Before creating the release tag, invoke the **Document Generator** with the "Updating docs" workflow. This combines **both** discovery methods to ensure nothing is missed:

1. **Pending notes** — Doc Generator collects all `.ai-docs/pending-doc-updates.md` files from recently merged branches (high-signal, structured)
2. **Git diff** — Also runs `git diff --name-only <last_tag>..HEAD` to catch any changes not covered by pending notes (safety net)
3. Merges and deduplicates both sources into a single change list
4. Reads the current shared docs and the source files referenced
5. Updates CODEBASE_MAP, GLOSSARY, BUILD_GUIDE, AGENT_GUIDE, ARCHITECTURE docs as needed
6. Updates `<!-- last-updated -->` dates on all changed docs
7. Runs conformance checks on all updated docs
8. Commits as: `docs(ai-agents): batch doc update for <TAG_NAME>`
9. Tag after docs are committed

### What Goes Where

| Doc Type | In Feature Branch | At Tag Time |
|---|---|---|
| **New feature docs** (new files) | Create immediately | Doc Generator verifies conformance |
| **CODEBASE_MAP** | Pending note only | Batch update |
| **GLOSSARY** | Pending note only | Batch update |
| **Root ARCHITECTURE** | Pending note only | Batch update |
| **BUILD_GUIDE** | Pending note only | Batch update |
| **AGENT_GUIDE** | Pending note only | Batch update |

### Why Per-Tag

- **Zero merge conflicts** — one writer, one branch, one commit
- **Docs stay accurate per release** — every tagged version has correct docs
- **Low overhead during dev** — developers just note what changed, not rewrite docs
- **Reviewable** — the batch doc update is a single PR that can be reviewed
