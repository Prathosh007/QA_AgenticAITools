How to Use — Step by Step

The Testcase Generator runs as an AI chat agent on the web at https://prathosh-14802-t. No installation needed — just open your browser, sign in, and start generating.

1

Open the Web Tool

Navigate to the UEMS Agent AI Toolkit in your browser:

🌐 https://prathosh-14802-t/explorer/

Sign in with your GitHub account (requires GitHub Copilot license). The tool authenticates via GitHub Device Flow — click "Continue with GitHub", enter the code shown, and authorize.

2

Select the AI Agent

In the bottom-right of the chat input area, click the agent selector dropdown to choose your agent:

☑️ Testcase Generator
📦 GOAT JSON Converter

Testcase Generator (default) — generates test cases AND converts to GOAT JSON in one flow.
GOAT JSON Converter — converts existing test cases to GOAT JSON payloads only.

💡 Tip: You can also select the AI model (Claude Sonnet 4.6 recommended) from the model dropdown next to the agent selector.
3

Fill in the Input Wizard

When the Testcase Generator agent is selected, a structured input form appears automatically — no need to type a free-text prompt. Fill in the fields:

✏️
Functionality Name e.g. "Agent Installation", "Computer Rename", "Software Deployment"
🔀
Generation Mode Choose Functionality-level (from scratch) or Diff Comparison (branch diff)
🌿
Diff Comparison fields Source branch, target branch, and repository name — AI fetches the diff automatically
📦
Functionality-level field Repository name where the functionality code lives — AI reads it to ground test cases in real code
💡 Tip: The repository field is mandatory in both modes. Providing the exact repo name ensures the AI searches the right codebase and generates accurate, grounded test cases. Common choices: uems_win_agent_setup (Windows), uems-mac-agent-setup (macOS), dc_native (Linux).
4

AI Processes Your Request Automatically

Once you hit Send, the AI agent runs through all phases automatically — no further input needed (unless it asks a clarifying question):

🏥
Phase 0 — DB Health Check Verifies the testcase DB is online before starting
🔍
Phase 1 — Collect Inputs Asks for mode, functionality, and platform if not provided
⚙️
Phase 2 — Generate Test Cases Reads code, knowledge files, and generates exhaustive TCs
📦
Phase 3 — GOAT Conversion Automatically converts all TCs to GOAT JSON payloads

You'll see real-time progress in the chat as the AI searches code, generates tests, and processes GOAT conversions. Look for structured log blocks (📂 REPO LOG, 📖 KNOWLEDGE LOG, etc.) for traceability.

5

View Results in the Testcase DB Viewer

After generation completes, all test cases and GOAT payloads are saved to the central database. View them at:

📊 https://prathosh-14802-t/tc/view
📋
Test Cases tab Browse all generated test cases — filter by functionality, priority, category
📦
GOAT Payloads tab View converted JSON payloads — copy payload JSON for each test case
⚠️
Gap Reports tab See which steps couldn't be mapped to GOAT operations with suggestions
📥
Export to CSV Download all test cases as a CSV file for spreadsheet use
💡 Tip: The dashboard cards at the top show Total TCs, Converted (GOAT), Unconverted, and Gap counts. Hover on the Gaps card to see a breakdown by missing utility.
6

Access Output Files

Besides the DB, the AI also saves files locally on the server workspace:

Phase Details

Phase 0 DB Health Check

  • Read testcases/db-config.json to get DB URL
  • Call GET /health on the testcase DB server
  • If offline → abort and show instructions to start the server
  • If online → print existing testcase and GOAT payload counts

Phase 1 Collect Inputs (Wizard)

  • Functionality name: The feature or module to test
  • Mode: Functionality-level or Diff comparison
  • Repository (both modes): Where the functionality code lives — mandatory for accurate code grounding
  • Diff mode: Source branch → target branch
  • Platform: Windows, macOS, or Linux

Phase 2 Generate Test Cases

  • Check DB for existing test cases — skip duplicates
  • Start with the user-provided repository → expand to dependencies via repos.json
  • Search codebase for entry points, data flows, config keys, error paths
  • Read KNOWLEDGE.md, SUPPORT_TICKETS.md, CSV files
  • Generate exhaustive TCs across 18 scenario categories
  • Run quality checklist → write CSV → POST to DB → save backup

Phase 3 GOAT JSON Conversion

  • Read GOAT_Operations_Context.md — sole authoritative source
  • Map each test step to a GOAT operation_type + action
  • Fully mappable TCs → JSON payload saved to goat-payloads/
  • Unmappable steps → gap report with suggested GOAT additions
  • POST payloads and gaps to testcase DB

Phase 4 Summary Report

  • Write summary.md — Markdown with tables and metrics
  • Write summary.html — self-contained styled dashboard
  • Print final summary to chat: counts, file paths, DB links
