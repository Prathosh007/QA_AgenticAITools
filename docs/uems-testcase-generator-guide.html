🧪 UEMS Testcase Generator

A complete guide to how test cases are generated from your codebase, converted into GOAT automation payloads, and tracked in a persistent database.

🎯 What Is This?

The UEMS Testcase Generator is an AI-powered agent that automates the entire QA workflow:


You say: "Generate test cases for Computer Rename"
It produces: structured manual test cases + GOAT automation JSON + a summary report — all in one run.

📋
Test Cases
Hierarchical manual TCs with product-level steps
⚙️
GOAT JSON
Automation payloads ready for execution
💾
Persistent DB
Never duplicates — reuses & extends existing TCs
⚠️
Gap Reports
Unmappable steps are flagged, never silently skipped

🔀 Two Generation Modes

📋 Functionality-Level

Generate test cases for a feature or module from scratch (or extend existing coverage).

  • Reads KNOWLEDGE.md for product context
  • Searches codebase for real file paths & configs
  • Covers: Happy Path → Failure → Edge → Negative → Integration

Example prompts:

  • "Generate test cases for Computer Rename"
  • "Generate test cases for Software Deployment > Patch Installation"

🔃 Diff Comparison

Generate test cases based on what changed between two branches.

  • Diffs source branch vs target branch
  • Classifies each change by functionality
  • Generates TCs only for affected areas

Example prompts:

  • "Generate diff test cases: feature_win_26.05 vs main"
  • "What changed between feature_rename_fix and main?"

How Are Relevant Repos Identified?

💡
Not all 38 repos are cloned. Only the repos relevant to your task are pulled — through a 3-layer identification process.
1
AI Reads Your Request
You say "Generate test cases for Patch Management on macOS". The AI matches keywords against repo descriptions and repo-map.md guidelines.

Result: Primary repo identified → patch-management
2
Dependency Chain Is Resolved
repos.json says patch-management depends on:
  • agent-utils (shared wrappers)
  • cmickey_utils (legacy utilities)
  • uems_native_dependencies (third-party libs)
The system recursively walks the dependency tree so nothing is missed.
3
On-Demand Clone
The setup-workspace tool looks up each repo's gitUrl in repos.json and runs git clone (or git fetch if already present).

Only the needed repos are cloned — the AI then searches this local code to understand real file paths, configs, and behaviors for generating accurate test cases.
⚡
Key insight: It's the AI's reasoning that picks the relevant repos. repos.json provides the metadata — the AI does the thinking. If it can't determine which repos, it simply asks you.

🔄 End-to-End Workflow

Here's everything that happens when you ask the Testcase Generator to produce test cases — from the moment you type the prompt to the final report.

🏥 Phase 0 — Database Health Check

Before anything else, the agent checks if the testcase database is running:

GET http://localhost:3000/health // If online → shows stats (total TCs, converted, gaps) // If offline → stops and tells you how to start the server
⛔
The generator will not proceed until the DB is reachable. This prevents duplicate test cases.

🧪 Phase 2 — Generate Test Cases

The testcase-generation skill runs a systematic process:

StepActionDetails
1 Check existing DB Queries the database to find existing test cases for this functionality — avoids duplicates
2 Gather context Functionality mode: reads KNOWLEDGE.md, existing CSVs, and searches the codebase
Diff mode: runs uems_agent_diff_branches to get actual code changes
3 Classify & categorize Groups code changes by functionality, sub-functionality, platform, and complexity
4 Generate test cases Creates TCs in priority order:
Happy Path Bug Scenario Failure/Retry Edge Case Negative Integration
5 Write product-level steps Steps use tester language (not code references) — real file paths, service names, and GOAT hints
6 Save to CSV & DB Outputs CSV file + POSTs each TC to the database
✅
Smart merge: If test cases already exist, the system merges — modifying outdated ones, adding new ones, and deprecating obsolete ones. Never starts from scratch unnecessarily.
▼
⚙️ Phase 3 — Convert to GOAT JSON

The testcase-json-converter skill automatically converts each test case into an executable automation payload:

StepActionDetails
0 Read GOAT operations Loads GOAT_Operations_Context.md — the single source of truth for all 22+ operation types
1 Skip already converted Checks DB: if a payload already exists for a TC → skips it
2 Map steps → operations Each test step is matched to a GOAT operation_type + action + parameters
3 Build JSON payload Assembles the full payload with component, operations array, and expected result
4 Save or report gap Mappable → saved to DB & file
Unmappable → gap report with missing util name & suggestion

❓ Frequently Asked Questions

Does it pull all 38 repositories?

No. Only repos relevant to your task are cloned on-demand. The AI identifies which repos are needed from your prompt, then repos.json provides the Git URLs and dependency chain. If you're testing "Computer Rename", it only clones the repos that contain computer rename logic and their dependencies.

What if the testcase DB is offline?

The generator stops at Phase 0 and tells you exactly how to start it:

cd source/testcase-db && node server.js

This is mandatory to prevent duplicate test cases across runs.

What if a test step can't be automated with GOAT?

It's never silently skipped. The converter creates a gap report entry with:

• The test case ID
• The unmappable step text
• The missing GOAT operation name
• A suggestion for what operation should be added to GOAT

What happens if I run it again for the same functionality?

It merges intelligently. Existing test cases are loaded from the DB, then:

• New scenarios → added with new IDs
• Changed scenarios → updated in place
• Obsolete scenarios → deprecated (not deleted)
• Unchanged → left as-is

What is GOAT?

G.O.A.T = Generic Orchestrated Automated Testing. It's the testing framework used by UEMS that executes JSON-defined test operations on remote machines. Each operation has a type (like registry_operation, service_actions, file_folder_operation) with specific actions and parameters.

Can I use this without the Orchestrator?

Yes. The Testcase Generator is a standalone agent. You can invoke it directly in VS Code Copilot Chat. It handles repo cloning and context gathering on its own — the Orchestrator is only needed for full SDLC workflows (plan → build → review → test).
