02 · How the Codebase Is Connected

UEMS agent code lives across ~38 repositories (macOS, Linux, Windows). A registry file called repos.json acts as a phone book — it stores metadata about each repo, not actual source code.

What's in repos.json?

FieldPurposeExample
gitUrlGit clone URL for the repohttps://git.csez.…/patch-management.git
layerBuild dependency order (0 = no deps)3
dependenciesOther repos this one depends on["agent-utils", "cmickey_utils"]
platformTarget OSmac, linux, windows
descriptionWhat the repo does (used by AI)"Patch management functionality"

How Are Relevant Repos Identified? (3 Layers)

⚡
Not all 38 repos are cloned. Only repos relevant to your task are pulled — through a 3-layer process.
  • 1
    AI Reasoning — The agent reads your request, matches keywords against repo descriptions and repo-map.md guidelines. If unsure, it asks you. Result: primary repo(s) identified.
  • 2
    Dependency Chain — repos.json is walked recursively to discover all upstream repos the primary repo depends on. Nothing is missed, nothing extra is pulled.
  • 3
    On-Demand Clone — The setup-workspace tool looks up each repo's gitUrl in repos.json and runs git clone (or git fetch if already present). Local code is then searched for real file paths, service names, and configs that go into test steps.

03 · Two Generation Modes

📋 Functionality-Level Mode

Generate test cases for a feature or module — from scratch or extending existing coverage.

  • Reads KNOWLEDGE.md for product context
  • Searches codebase for real paths & configs
  • Covers all scenario categories

Example:
"Generate test cases for Computer Rename"

🔃 Diff Comparison Mode

Generate test cases based on what changed between two branches.

  • Diffs source branch vs target branch
  • Classifies each change by functionality
  • Generates TCs only for affected areas

Example:
"Diff test cases: feature_win_26.05 vs main"

Automated conversion of manual test steps into executable GOAT framework JSON payloads — with strict schema validation and named gap reporting for every unmappable step.

05 · GOAT JSON Converter — Deep Dive

G.O.A.T = Generic Orchestrated Automated Testing. It's the testing framework that executes JSON-defined operations on remote machines. The converter translates each manual test step into a typed GOAT operation — or flags it as a gap if no matching operation exists.

⚙️ Phase 3 — GOAT JSON Conversion (testcase-json-converter skill)
StepActionDetail
0Read GOAT_Operations_Context.md Sole source of truth for all operation types, actions, and parameters. Nothing is invented. If an operation isn't in this file, it goes to the gap report.
1Skip already converted GET /payloads/:id → 200 = skip · 404 = convert. Only new TCs are processed.
2Map steps → GOAT ops Each step mapped to operation_type + action + parameters. Exact names checked against the operations file.
3Auto-detect component Component is inferred from step context:
$LocalOffice_WinAgent1 — agent process/service/registry
$DS_Machine_IP — Distribution Server
$Server_Machine_IP — UEMS Server
4Build JSON payload Assembles operations array with component, expected result, and goatexecutewithpayload() call
5Save or report gap Mappable → POST /payloads + save to goat-payloads/
Unmappable → POST /gaps + named gap report

JSON Payload Structure

{ "TC-RENAME-001": { "testcase_id": "TC-RENAME-001", "product_name": "UEMS Agent", "reuse_installation": false, "operations": [ { "operation_type": "service_actions", "parameters": { "action": "stop", "service_name": "ManageEngine UEMS - Agent" } }, { "operation_type": "registry_operation", "parameters": { "action": "set", "path": "HKLM\\SOFTWARE\\UEMS Agent\\ComputerName", "value": "NEW-PC-NAME" } }, { "operation_type": "service_actions", "parameters": { "action": "start", "service_name": "ManageEngine UEMS - Agent" } } ], "expected_result": "Agent re-registers with the new computer name in the server console" } }

Mappability Decision

⚖️ Are All Steps Mappable to a GOAT Operation?
Each step checked against GOAT_Operations_Context.md — exact match only, no guessing
✅ Yes — All Mapped
JSON payload saved to DB via POST /payloads and written to goat-payloads/. Viewer shows Converted.
⚠️ No — Gap Found
Gap entry created via POST /gaps with missing util name and suggested operation. Viewer shows Unconverted.

06 · Available GOAT Operations

All valid operation_type values come exclusively from GOAT_Operations_Context.md. Common types used in UEMS test cases:

service_actionsStart / stop / status of Windows services
registry_operationRead / write / delete registry keys
file_folder_operationCheck file/folder existence
file_folder_modificationRead / write / edit config files
run_commandExecute system shell commands
run_batExecute .bat / .cmd scripts
exe_installRun installer executables
task_managerVerify process running or not
communication_operationHTTP request / response checks
download_fileDownload file to a path
gui_operationWeb UI browser automation
native_gui_operationNative Windows GUI (automationId)
⛔
Strict rule: The converter never invents an operation. If a step requires something not in GOAT_Operations_Context.md, it goes into the gap report — stating exactly which util is missing and what it should do.

07 · Gap Reports

A gap report is generated whenever a test case has one or more steps that cannot be mapped to any existing GOAT operation. The report is saved to testcases/<F>/goat-payloads/gap-report.md and also posted to POST /gaps.

FieldValue ExamplePurpose
tc_idTC-INSTALL-006Which test case has the unmappable step
missing_utilwindows_event_log_operationName of the GOAT operation that needs to be created
affected_step"Check Event Log for ID 7036"Exact step text that can't be automated
suggestion"Read Windows Event Log by source and event ID"Description of what the new operation should do
💡
The DB viewer tooltip shows un-converted TCs grouped by their gap reason when you hover over the Unconverted counter card. This helps prioritize which GOAT operations to build next.

FAQ

Does it pull all 38 repositories?
No. Only repos relevant to your task are cloned on-demand. The AI identifies primary repos from your prompt, then repos.json provides the Git URLs and dependency chain. A "Computer Rename" task only clones repos that contain rename logic plus their dependencies.
What if the testcase DB is offline?
Phase 0 blocks and tells you exactly how to start it: cd source/testcase-db && node server.js. This is mandatory — the DB tracks all existing TCs to prevent duplicates.
What if a test step can't be automated with GOAT?
It is never silently skipped. A gap report entry is created with: the TC ID, the unmappable step text, the missing operation name, and a suggested operation description. Posted to /gaps and saved to gap-report.md.
What happens if I run it again for the same functionality?
Smart merge applies: New scenarios → added with new IDs · Changed scenarios → updated in place · Obsolete scenarios → deprecated (not deleted) · Unchanged → left as-is. It never regenerates from scratch.
Why were the Unconverted TCs showing as a negative number in the viewer?
This was a race condition bug: loadPayloads() and loadStats() ran in parallel. When loadPayloads() finished first, the total count wasn't written to the DOM yet — so the formula total - converted evaluated as 0 - 37 = -37. Fixed by moving updateConversionCounts() to after all parallel loads complete.
