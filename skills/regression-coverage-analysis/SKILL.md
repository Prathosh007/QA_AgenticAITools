# Regression Coverage Analysis Protocol

Analyze code changes to identify existing functionality that needs retesting and gaps in test coverage. Produces a structured markdown summary for QA leads.

## When to Use
- After generating manual test cases for a diff (QA agent Step 8)
- When a reviewer identifies cross-component impact and wants retest guidance
- During release preparation to plan regression test scope
- When the Orchestrator or Delta Reviewer needs retest impact analysis

## When NOT to Use
- For generating executable test cases — use `manual-test-generation` skill instead
- For unit test scripts — use `unit-test-runner` skill instead
- For code review findings — use the Reviewer agent directly

---

## Core Principle

> **Trace impact, not assumptions.** Every retest area must trace to a specific code change. Do not list generic areas "just in case."

---

## Procedure

### Step 1: Identify Changed Modules

From the diff or changed file list, map each changed file to its module or functional area:

1. **Group files by module** — use directory structure, component names, or `repo-map.md` mappings
2. **Classify change type per module** — logic change, refactoring, config, dependency (same categories as the Reviewer's Change Summary)
3. **Note shared code** — files in utility/shared/common directories affect multiple consumers

### Step 2: Trace Shared Code Paths

For each changed file in a shared or low-layer location:

1. Use `uems_agent_dependency_graph` (direction: "down") to find downstream dependents
2. Use `uems_agent_search_repos` to find callers of changed functions
3. Map each caller to its functional area — these are retest candidates

**Shared code indicators:**
- Files in `Agent-Utils`, `CommonUtil`, `framework`, or similar shared directories
- Functions called from multiple modules (>1 caller in different directories)
- Config/constant changes that affect conditional behavior in multiple places
- Header file changes (struct definitions, enums, macros) included by multiple TUs

### Step 3: Identify Retest Areas

For each downstream dependent or caller found in Step 2, determine:

| Field | How to Determine |
|-------|-----------------|
| **Area** | Functional area name (e.g., "Agent Refresh / Slot Time", "Tray Icon Animation", "Certificate Chain Validation") — product-level, not code-level |
| **Why Retest** | What specific change could affect this area? Reference the changed file/function and explain the dependency. |
| **Suggested Tests** | Product-level test actions to verify the area still works. Same language rules as QA test steps — no code references. |

**Rules:**
- Every entry must have a concrete "why" — not "this area might be affected"
- Suggested tests must be specific enough to execute: name the artifact to check (log file, config, UI element, behavior)
- If the change is purely additive and does not touch existing code paths, state "No regression risk — additive only"

### Step 4: Identify Coverage Gaps

Scan for areas where the changes have **no existing test coverage**:

1. **No manual test cases** — the CSV generated by the QA agent doesn't cover this area (new module, new component, infrastructure code)
2. **No automated tests** — no unit tests or integration tests exist for the changed functions (check for `*_test.go`, test directories, test projects in the repo)
3. **Untestable without environment** — changes that require specific hardware, network conditions, or multi-machine setups that can't be easily reproduced

For each gap, state:
- What area lacks coverage
- Why it's a gap (no tests exist, tests can't cover this scenario, etc.)
- Impact if the gap is not addressed (silent regression risk level: HIGH / MEDIUM / LOW)

### Step 5: Generate Report

Produce the regression coverage summary in the format defined by `testing-standards.md` §7.

```markdown
# Regression Coverage Summary

**Source:** `<sourceBranch>`
**Target:** `<targetBranch>`
**Date:** <YYYY-MM-DD>

## Existing Functionality to Retest

| # | Area | Why Retest | Suggested Tests |
|---|------|------------|----------------|
| 1 | <functional area> | <what changed that could affect it> | <what to verify> |

## Missing Test Coverage

- <gap description> — Impact: HIGH / MEDIUM / LOW
```

**File naming:** `regression-coverage-<sourceBranch>-vs-<targetBranch>-<YYYY-MM-DD>.md`
Sanitize branch names: replace `/` with `-`.
**Output location:** Same directory as the test case CSV (workspace root).

## Quality Gate

The output must satisfy:
- Every retest area traces to a specific code change (no generic entries)
- Suggested tests use product-level language (no function names, no line numbers)
- Coverage gaps include impact assessment
- If changes are purely additive with no shared code impact, the "Existing Functionality to Retest" section states "No regression risk — additive only" (do not leave empty)
