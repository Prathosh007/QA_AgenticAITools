Problem Statement

In 2022, A bug-bounty case was received stating that our cloud's development build was exposed to the public github repository manually by a user. This data contains sensitive keys like Master Key, which is used to encrypt sensitive data across the product. Due to the above issue, the static encryption key (Master Key) used in cloud development setup was exposed. 


Fortunately, our production environment uses different keys than the development environment, so the impact of this case was reduced. However, it serves as a reminder that similar cases could occur in the future.


In the case of our On Premise solution, the presence of a single Master Key for encryption and decryption poses a significant risk. If a single setup is exploited, it could have a detrimental impact on all of our customers.


In 2026, around May last week, the OP Master Key has been exposed. We need to act fast in rotating the Master Key.


Proposed Solution

In order to mitigate the potential consequences of a compromised setup, we have decided for the implementation of module(s) or microservice specific encryption keys with a provision to regenerate the keys for each environment. In Cloud, we will be having the service level master keys unique for each customer (say organization). 


Will we share the service level master keys before the agent is approved ?

No, the service level master keys will be shared to the agent only after the agent is approved in the console. For any sensitive data that is shared to the agent that is awaiting approval, we will make use of the Approval Key for encyption/decryption, so that the service level master keys are not exposed to them before approval.


Existing Workflow prior to Approval Token

In our Server and Agent, we use a static Master Key and Server Installed Time as Salt to encrypt and decrypt sensitive data. Master Key is a secure secret key used for encrypting and decrypting sensitive data. This key has been the same for all the setups. 

In the current flow, the static master key is bundled with the agent binaries during build time.


Approval Token Workflow (Released)

Agents will be bundled with the approval key and approval token in the binaries instead of auth token and dynamic key so that the auth token will not be shared to the agent until it is approved. Once the agent is approved, the auth token will be shared.


Proposed Workflow

We will be reducing the dependency of the static master key module wise and will be shifting towards the service specific master key.

We already have our approval token flow that shares the auth token only after the agent is approved. Similarly, certain data that is being shared with the agents before approval has to be encrypted. Such data will be encrypted using APPROVAL KEY which will remain unique and static for each setup. Only after the agent is approved we will share the service level master keys to the agent via KeyShareServlet which will return the module wise keys as requested by the agent.

As of the first release, we will be proceeding only with SOM_MASTER_KEY where the som files dependent on the master key will be decrypted with the static master key and re-encrypted with the new master key.

Key Regeneration

We will be introducing a provision to regenerate the module wise master keys so that in case of any compromise, the service level master keys can be regenerated for a setup immediately. During this regeneration, we need to make sure there won't be any client side API calls, or agent requests, or tasks or schedulers running. Hence the regeneration workflow will be triggered only in the maintenance window.

During the maintenance window, all the agent requests will be responded with a specific CODE:__________ , so that the agent shall contact the server after 90 mins or only at the next refresh cycle. 

We will be halting all the schedulers as a pre-requisite, and when the regeneration flow is completed, we will be enabling them back. Tasks scheduled at the downtime/maintenance window will be triggered automatically once enabled back.

Each module or a group of modules or a microservice will have their own master key and is independent of the other module(s) or app services. The framework for stopping the schedulers and re-enabling them back will be provided. But the service level rotation implementation has to be taken care of by the module(or microservice) only

For the above encrypt and decrypt native functions developed in this topic, the salt will be passed from server side instead of the working directory(in the old functions).

In case of the key being passed as null in the encryptWithKeyAndSalt(), decryptWithKeyAndSalt(), encryptWithKey(), decryptWithKey() -> we will use the static master key(bundled key) for the util. 


encryptWithKey(plaintext, key) and decryptWithKey(cipherText, key)

These functions will encrypt and decrypt the text with the key shared. If the key is null -> static master key will be used.

 
 encryptWithSalt(plainText, salt) and decryptWithSalt(cipherText, salt)

These functions will encrypt and decrypt the text with the salt value shared in param. Salt value will always be passed from server to native function. In this util no need default salt handling, since this is introduced as new.

 
encryptWithKeyAndSalt(plainText, key, salt) and decryptWithKeyAndSalt(cipherText, key, salt)

These functions will encrypt and decrypt the text with the key and salt shared in params. If the key shared is null, then we make use of the static master key. Salt will always be passed as param from server. No need default handling for salt in this util.

In OP,
Agent Binaries will contain Approval Token and Approval Key in the json. Approval Token will be encrypted with Approval Key and Approval Key will be encrypted with SALT. For CSR request, the request and response shall be encrypted and decrypted with Approval Key instead of Master Key. Old Agents shall get the data in the old workflow itself. 

Once the agent hits /AuthProps POST,  the agent gets added in waiting for approval and gets responded with Approval Token (encr with Approval Key) and Approval Key (encr with SALT). Once the agent is approved / auto approved,  the /AuthProps GET responds with the Auth token(encr with Dynamic Key) and Dynamic Key(encr with Approval Key) is shared to the agent. With the auth token, the agent shall hit the /KeyShareServlet and get the services master key, which has to be decrypted with the Dynamic Key.


In Cloud,

Agent Binaries will contain Provisional Key and RemoteOfficeAuthKey Approval Key in the json. Provisional Key will be encrypted with Approval Key and Approval Key will be encrypted with SALT. For CSR request, the request and response shall be encrypted and decrypted with Approval Key instead of Master Key. Old Agents shall get the data in the old workflow itself. 

Once the agent hits /AuthProps POST,  the agent gets added in waiting for approval and gets responded with Approval Token(encr with Approval Key) and Approval Key. Once the agent is approved / auto approved, the /AuthProps GET responds with the Auth token(encr with Dynamic Key) and Dynamic Key(encr with Approval Key) is shared to the agent. With the auth token, the agent shall hit the /KeyShareServlet and get the services master key, which has to be decrypted with the Dynamic Key.


As of in the first release, we will be introducing a fw for the master key rotation and the static master key(existing) dependency in som will be uncoupled and migrated to the som service master key.

In this case, 

1. New exe customers will get new service level keys from the build installation startup. 

2. Existing customers via PPM upgrade, shall have the existing master key only in the new workflow for the first time alone. Next re-generation will share a new unique key for each of the services registered in the framework.
