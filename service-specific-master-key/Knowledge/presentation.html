Service Specific Master Key (SSMK)

Replacing static Master Key with per-service dynamic encryption keys

Contents

1. Problem Statement 2. Solution Overview 3. Encryption Key Mapping (Before vs After) 4. Authentication Flow 5. KeyShareServlet API 6. Key Regeneration 7. Native Encrypt/Decrypt Functions 8. Implementation Changes

1. Problem Statement

Incident: Cloud development build was exposed to public GitHub via bug-bounty. Static Master Key was leaked.
  • Production uses different keys — impact was limited this time
  • On-Premise risk: Single static Master Key for ALL setups → one compromise = all customers affected
  • Master Key bundled directly in agent binaries during build

2. Solution Overview

Introduce module/microservice-specific encryption keys with regeneration capability. Cloud keys will be unique per organization.
  • Service level master keys shared only after agent approval
  • Pre-approval data encrypted with Approval Key (unique per setup)
  • Keys fetched via KeyShareServlet post-approval
  • Each module/microservice has an independent key

4. Authentication Flow

On-Premise (OP)

StepActionData
1 Agent installs (binary contains) Approval Token (encr Approval Key) + Approval Key (encr Salt)
2 /AuthProps POST Added to waiting-for-approval. Response: Approval Token + Approval Key
3 /AuthProps GET (after approval) Auth Token (encr Dynamic Key) + Dynamic Key (encr Approval Key + Salt)
4 /KeyShareServlet POST Service master keys (decrypt with Dynamic Key)
Backward Compat: New agents pass APIVersion=v1.1 query param. Old agents continue with static master key during 30-day transition.

Cloud

StepActionData
1 Agent installs (binary contains) Provisional Key (encr Approval Key) + Approval Key (encr Salt)
2 /AuthProps POST (awaiting approval) Approval Token (encr Approval Key) + Approval Key (encr Salt)
3 /AuthProps GET (once approved) Auth Token (encr Dynamic Key) + Dynamic Key (encr Approval Key + Salt)
4 /KeyShareServlet POST Service master keys per org (decrypt with Dynamic Key)

DS → Provisional Key acts as Auth Token directly. | WFA disabled: POST gives Auth Token, action=11 returns Approval Token+Key.

5. KeyShareServlet API

POST /servlets/KeyShareServlet — Agent sends key UUIDs, receives module keys

Request

{
  "moduleKeyInfo": [
    { "keyUUID": "fda5087a-...-688" },
    { "keyUUID": "148c29ca-...-189" }
  ]
}

Response (decrypted)

{
  "serviceMKeys": [
    {
      "isKeyAtAdaptionStage": false,
      "moduleName": "SOM",
      "serviceKey": "24fe22...",
      "keyUUID": "fda5087a-..."
    }
  ]
}

Response Status

moduleNameMeaningAgent Action
Valid (SOM, CRED...) Key shared successfully Use serviceKey for encrypt/decrypt
Invalid UUID not valid for this server Ignore
NotApplicable Agent type doesn't need this key Do NOT re-request even after rotation

isKeyAtAdaptionStage = true: Module onboarded but still uses old bundled master key as service key (transition state).

7. Native Encrypt/Decrypt Functions

// Key + Salt
public native String encryptWithKeyAndSalt(String plainText, String key, String salt);
public native String decryptWithKeyAndSalt(String cipherText, String key, String salt);

// Key only (if key=null → falls back to static master key)
public native String encryptWithKey(String plainText, String key);
public native String decryptWithKey(String cipherText, String key);

// Salt only (salt always from server, no default handling)
public native String encryptWithSalt(String plainText, String salt);
public native String decryptWithSalt(String cipherText, String salt);
FunctionKey BehaviorSalt Behavior
encryptWithKeyAndSalt / decryptIf null → static master keyAlways from server param
encryptWithKey / decryptIf null → static master keyN/A
encryptWithSalt / decryptN/AAlways from server param (new, no default)
