Feature Summary

SSMK replaces a single shared master key with per-service-module encryption keys (AGENT_UTILS, FRAMEWORK, PATCH, INVENTORY, DCP, ACP, DMS, SOM). Keys are negotiated/rotated with the DS via a Key Share servlet and stored in a per-platform vault (Win registry + AES blobs, Linux SQLite, Mac plist + keychain). Auth tokens, proxy creds, cert keys are all re-encrypted under the per-module key. Lifecycle states: Active → Adaption → Active with both keys valid during adaption window.

High-Risk Areas

R1 — Credential/key encryption. Wrong key = permanent data loss. Affects all 7 repos.
R2 — Key Share handshake. Network race, 401/403/503 from DS, partial response handling. Watch TC-SSMK-KEYSHARE-*.
R3 — Upgrade migration. One-shot migration; failure leaves agent unable to decrypt. Old-master-key file must be removed only after vault verified. TC-SSMK-UPGRADE-*, TC-SSMK-LIN-UPGRADE-*.
R4 — AES-GCM nonce reuse. Three independent implementations added across repos. Catastrophic if nonce reused with same key. TC-SSMK-LIN-VAULT-003.
R5 — DS-DS replication. Two DS rotating same module simultaneously. TC-SSMK-DSREPL-003.
R6 — Backward compatibility. New agent ↔ old DS and old agent ↔ new DS must both work or fail safely. TC-SSMK-INT-AGENTDS-002/003.
R7 — Mac DMS WebSocket revamp. Bundled into framework-ops-suite — independent risk vector. TC-SSMK-MAC-DMS-*.
